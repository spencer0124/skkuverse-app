# UX 라이팅 가이드

TDS 공식 UX 라이팅 가이드 + 토스 8가지 라이팅 원칙 기반.
사용자에게 노출되는 모든 문구에 적용할 것.

---

## 코어밸류 (보이스)

| 가치 | 의미 |
|------|------|
| **Clear** (명확한) | 쉽고 명확한 단어, 한 번에 이해되는 문장 |
| **Concise** (간결한) | 꼭 필요한 정보만, 빠르게 스캔 가능하도록 |
| **Casual** (친근한) | 어려운 용어/딱딱한 뉘앙스 대신 일상 언어 |
| **Respect** (존중하는) | 전환율 욕심보다 신뢰 우선, 진정성 있는 커뮤니케이션 |
| **Emotional** (공감하는) | 사실 전달 + 감정 공감 |

## 필수 규칙

### 1. 해요체 통일

모든 문구를 해요체로. 상황/맥락 불문.

```
✕ 운행 안함
✕ 운행하지 않습니다
✓ 운행하지 않아요
```

### 2. 능동형 문장

"~됐어요" 대신 "~했어요". "~었" 빼기. 동사 바꿔쓰기.

```
✕ 업데이트가 완료됐어요
✓ 시간표를 업데이트했어요

✕ 저장되었어요
✓ 저장했어요

✕ 발송됐어요
✓ 보냈어요
```

### 3. 긍정형 문장

"안 돼요" 대신 "~하면 할 수 있어요".

```
✕ 현금 결제 불가
✓ 카드나 T머니로 결제할 수 있어요

✕ 운행중인 버스 없음
✓ 오늘 운행이 끝났어요 (+ 내일 첫차 안내)

✕ 검색 결과 없음
✓ 찾는 건물이 없어요. 다른 이름으로 검색해 보세요
```

### 4. 캐주얼 경어

"~시겠어요?", "~시나요?", "~께" 같은 과도한 경어 금지.

```
✕ 검색하시겠어요?
✓ 검색해 보세요

✕ 어떤 건물을 찾고 계신가요?
✓ 어떤 건물을 찾고 있나요?

✕ 이름이 어떻게 되시나요?
✓ 이름이 뭐예요? (또는: 이름을 알려주세요)
```

경어를 뺐을 때 어색하면 **파악하고 싶은 정보를 주어로** 문장을 새로 쓰기.

### 5. 명사+명사 풀기

한자어 명사 → 동사 형태로 풀어쓰기.

```
✕ 시간표 업데이트
✓ 시간표가 바뀌었어요

✕ 서비스 점검
✓ 서비스를 점검하고 있어요

✕ 네트워크 오류
✓ 네트워크에 문제가 생겼어요
```

한자어 풀기 어려울 때: '{명사}가 {명사}해서' 형태로.

### 6. "되어요" → "돼요"

모바일 공간 절약. 모든 "되어요"를 "돼요"로.

```
✕ 적용되어요
✓ 적용돼요
```

## 8가지 라이팅 프린시플

### 1. Weed Cutting (잡초 제거)
의미 없는 단어 제거.

```
✕ 혹시 다른 건물을 찾고 있나요?
✓ 다른 건물을 찾고 있나요?

✕ 지금 바로 확인하세요
✓ 지금 확인하세요 ("지금"과 "바로" 의미 중복)
```

### 2. Remove Empty Sentences (빈 문장 제거)
화면에 불필요한 설명 문장 제거.

```
✕ 아래 목록에서 원하는 건물을 선택해주세요
✓ (목록만 바로 보여주기)

✕ 여기에서 셔틀 시간표를 확인할 수 있어요
✓ (시간표를 바로 보여주기)
```

### 3. Focus on Key Message (핵심 메시지 집중)
화면 간 중복 제거. 한 화면에서 전달하는 메시지는 하나.

### 4. User-side Info (유저 관점 정보)
시스템 관점이 아닌 사용자 관점으로 정보 제공.

```
✕ 서버에서 데이터를 불러오지 못했어요
✓ 건물 정보를 불러오지 못했어요

✕ GPS 신호가 약해요
✓ 현재 위치를 찾지 못했어요
```

### 5. Next Step Hint (다음 화면 힌트)
탭/클릭 후 뭐가 나올지 예상 가능하게.

```
✕ 더보기
✓ 층별 공간 보기

✕ 확인
✓ 시간표 보기
```

### 6~8. (위 규칙들의 심화)

## 예외 규칙

### 수동형 OK

- **서비스 종료/만료**: "이번 학기 운행이 종료돼요" (주어 강조)
- **인과관계 설명**: "카드를 등록하지 않으면 결제가 안 돼요"
- **안심 문구**: "개인정보는 수집되지 않아요"
- 단, 주기적 반복 종료에는 "종료돼요" 쓰지 않음 ("오늘 운행이 끝났어요" 사용)

### 경어 OK

- **맥락 추정 질문**: "혹시 비밀번호를 잊으셨나요?"
- **선의 필요 시**: "잠깐 설문에 참여해 주시겠어요?"

### 부정형 OK

- **정책상 불가**: "이 기능은 학생 인증 후에 쓸 수 있어요" (이유 함께 안내)
- **일부 기능 제한**: "현금으로는 결제할 수 없어요"
- **안심**: "위치 정보는 저장하지 않아요"

## 스꾸버스 문구 패턴

### 상태 메시지

| 상황 | 문구 |
|------|------|
| 셔틀 운행중 | `다음 버스 {시간}` |
| 셔틀 운행 종료 | `오늘 운행이 끝났어요` |
| 셔틀 운행 전 | `첫차 {시간}에 출발해요` |
| 셔틀 운행 없는 날 | `오늘은 쉬는 날이에요` |
| 로딩 중 | (스켈레톤 UI, 텍스트 없음) |
| 데이터 로드 실패 | `{대상}을 불러오지 못했어요` + `다시 시도해 주세요` 버튼 |
| 검색 결과 없음 | `찾는 건물이 없어요` |
| 빈 목록 | `아직 {대상}이 없어요` |

### 에러 메시지 템플릿

```
[제목]: {대상}을 불러오지 못했어요
[설명]: 잠시 후 다시 시도해 주세요
[버튼]: 다시 시도 / 닫기
```

- "닫기"를 왼쪽에, 확인/액션을 오른쪽에 배치.
- "취소" 대신 항상 "닫기" 사용.

### 셔틀 관련

```
✕ 운행시간 월요일 ~ 금요일
✓ 평일에만 운행해요

✕ 공휴일 운행 안함
✓ 공휴일은 쉬어요

✕ 요금 및 결제 400원
✓ 셔틀버스 요금 · 400원

✕ 후불교통결제 기능 가능한 카드에 한함
✓ 후불교통결제 가능한 카드만 돼요

✕ 금요일에는 기존 인자셔틀 버스와 별도로 학부대학 셔틀 버스가 추가운영됩니다
✓ 금요일엔 학부대학 셔틀이 추가로 다녀요

✕ 시간표 · 총 7편
✓ 하루 7회 운행
```

### 건물/공간 관련

```
✕ 건물 5건, 공간 20건
✓ 건물 5곳 · 공간 20곳

✕ 호실 4개
✓ 4개 공간

✕ + 6개 더보기
✓ 6곳 더 보기

✕ → 수선관(별관)
✓ 수선관 별관으로 연결돼요
```

## 스낵바/토스트 메시지

| 상황 | 문구 |
|------|------|
| 링크 열기 실패 | `링크를 열 수 없어요` |
| 네트워크 끊김 | `네트워크에 문제가 생겼어요` |
| 복사 완료 | `복사했어요!` |
| 이메일 복사 | `이메일 주소를 복사했어요` |

- 스낵바 제목에 **"오류" 사용 금지**. 무슨 일이 있는지 직접 설명.
- 본문에 사용자가 할 수 있는 행동 안내 (`잠시 후 다시 시도해 주세요`).

## 권한 요청 다이얼로그

| 요소 | 문구 |
|------|------|
| 제목 | `{권한}이 필요해요` |
| 본문 | `{기능}을 사용하려면\n{권한}을 허용해 주세요` |
| 왼쪽 버튼 | `닫기` ("취소" 사용 금지) |
| 오른쪽 버튼 | `설정으로 이동` (Next Step Hint) |

## "없음" 상태 규칙

```
✕ {정보} 없음     (명사+명사)
✓ {정보}가 없어요  (해요체 문장)
```

단, 극히 좁은 UI 공간(뱃지, 테이블 셀)에서는 짧은 라벨 허용: `없음`

## 다크패턴 방지

TDS 다크패턴 방지 정책에서 금지하는 것:

1. **진입 직후 바텀시트 팝업 금지** — 사용자가 원하는 화면을 먼저 보여줄 것
2. **뒤로가기 시 이탈 방지 팝업 금지** — 알림 동의 유도 등
3. **거절 선택지 없는 CTA 금지** — 항상 닫기/거절 옵션 제공
4. **예상 못한 광고 노출 금지** — 사용 흐름 중 갑작스러운 인터럽트 금지
5. **모호한 CTA 텍스트 금지** — 버튼 누르면 뭐가 되는지 명확히

## 그래픽 사용 규칙

1. **문맥에 맞는 그래픽** 사용 (장식 아님, 이해를 돕는 역할)
2. **한 화면에 핵심 그래픽 하나만** (나머지는 아이콘으로 대체)
3. **핵심 정보를 가리지 않게** 배치
4. **부정적/호소 감정 표현 금지**
5. **장식적 이펙트 금지** (의미 없는 파티클, 과한 그라데이션)
6. **상황을 정확히 전달** (오류 아닌데 느낌표 아이콘 쓰지 않기)
7. **아이콘 크기**: 24~40px
8. **아이콘 병렬 조합 금지**: 한 번에 하나만
