# Toss UI/UX 가이드

> 앱인토스(Apps-in-Toss) 개발자센터의 UI/UX 가이드를 정리한 문서예요.
> SDS 컴포넌트 개발 시 토스의 디자인 철학과 UX 원칙을 참고할 수 있어요.
>
> 출처: https://developers-apps-in-toss.toss.im/design/overview.html

---

## 1. Toss UX 원칙 (Product Principles)

토스의 모든 UI/UX는 아래 4가지 원칙을 따라야 해요.

| Principle | 적용 기준 | 예시 |
|---|---|---|
| **Simplicity** | 명료해야 하며, 추가 설명 없이 의미를 이해할 수 있어야 함 | "지금 보기", "광고 보기" 등 명확 CTA 사용 |
| **Clear Action** | 클릭 후 어떤 행동이 발생할지 사용자가 예측 가능해야 함 | 리디렉션·새창 이동 시 고지 문구 노출 |
| **No Deception** | 예상치 못한 순간, 형태, 위치에서 등장하지 않아야 함 | 서비스 진입 직후 전면 배너 금지 |
| **Value First** | 고객의 서비스 목표를 방해하지 않아야 함 | 결제/계좌 개설 흐름 중 광고 삽입 금지 |

## 2. UX 라이팅 가이드

토스 앱의 보이스톤을 적용한 문구 작성 지침이에요.

### 2.1 해요체

제품 안의 모든 문구는 **'해요체'**로 써요.
상황, 맥락을 불문하고 모든 문구에 해요체를 적용해요.

### 2.2 능동적 말하기

최대한 **능동형 문장**을 써요.

- **됐어요 → 했어요**
  - X: "송금이 완료됐어요" → O: "송금을 완료했어요"
- **'~었' 빼기**
  - X: "충전되었어요" → O: "충전했어요"
- **동사 바꿔쓰기**
  - 수동적 동사를 능동적 동사로 교체

### 2.3 긍정적 말하기

부정적 커뮤니케이션을 최대한 줄이고 긍정형 문장을 써요.

- **없어요 → 있어요**
  - X: "잔액이 부족해요" → O: "0원 이상 있으면 송금할 수 있어요"
- **에러 메시지**: 해결 방법을 함께 안내
- **다이얼로그 왼쪽 버튼은 항상 [닫기]** — "취소"는 사용자가 작업이 취소된다고 오해할 수 있어 쓰지 않아요.
- **혜택을 받을 수 없을 때**: 긍정형으로 전환

### 2.4 캐주얼한 경어

과도한 경어를 쓰지 않아요. 최대한 캐주얼하고 친근한 말투를 써요.

- **동사에서 '~시' 빼기**: "보내시겠어요?" → "보낼까요?"
- **'계시다' → '있다'**
- **'여쭈다' → '확인하다, 묻다'**
- **'께' → '에게'**
- 경어를 뺐을 때 어색하면 → 파악하고 싶은 정보를 '주어'로 써서 문장을 새롭게 작성

### 2.5 '{명사} + {명사}' 쓰지 않기

- **한자어 풀어쓰기**: 한자어 명사를 풀어서 동사 형태로 씀
  - X: "결제 취소" → O: "결제를 취소했어요"
- 풀어쓰기 어려울 경우: '{명사}가 {명사}해서' 형태로 풀어줌

### 2.6 예외 규칙

#### 수동형 문장을 써도 되는 경우

- **서비스 종료, 기간 만료**: 주어를 강조하고 '종료'와 '만료'의 뉘앙스를 정확히 전달
  - 주기적으로 종료가 반복되는 제품에는 '종료돼요'를 쓰지 않음
- **사용자에게 미치는 영향을 알려줄 때** (연체, 해지, 적용 등): 인과 관계를 명확하게 설명
- **사용자 안심**: 민감한 상황(정보 수집 안내 등)에서 안심시킬 때

> 참고: '되어요'는 모두 **'돼요'**로 통일

#### 경어를 써도 되는 경우

- **사용자의 맥락을 활용해서 질문할 때**: '시나요?', '셨나요?'로 당황스러움을 줄임
- **사용자의 상황을 추정할 때**: 직접 판단하게 해야 할 때 정중하게 질문
- **사용자의 선의가 필요할 때**: 설문조사 등

#### 부정형 문장을 써도 되는 경우

- **서비스를 정책상 쓸 수 없을 때**: 상황을 명확하게 인지시키고 이유를 함께 안내
- **일부 기능만 쓸 수 없을 때**: 어떤 기능을 쓸 수 없는지 명확히 전달
- **사용자 안심**: 민감한 상황에서 "~하지 않아요"로 안심시킬 때

## 3. 다크패턴 방지 정책

UX 가이드라인은 사용자에게 예측 가능하고 편리한 경험을 제공하기 위한 최소한의 기준이에요.
아래 사례들은 치명적인 사용성 오류로, 이를 위반하면 출시할 수 없어요.

### 3.1 서비스에 진입하자마자 바텀시트가 뜨는 경우

사용자가 기대한 화면 대신 **전면을 가로막는 광고성 바텀시트**가 노출되는 경우.
알림 동의를 요청하는 바텀시트도 포함.

> 사용자는 서비스에 진입한 순간, 자신이 의도한 목적을 바로 수행할 수 있기를 기대해요.

### 3.2 뒤로 가기 버튼을 눌렀을 때, 이전 화면을 막는 바텀시트가 뜨는 경우

이탈을 막기 위해 의도적으로 설계된 **알림 동의 유도 바텀시트** 등.

> 이탈 방지용 인터럽트는 사용자에게 자율성이 침해된다는 인상을 줄 수 있어요.

### 3.3 나갈 수 있는 선택지가 없는 경우

파트너사가 유도한 CTA를 선택하는 것 외에는 사용자가 **다른 선택을 할 수 없는 구조**.

> 거절할 수 없는 설계는 사용자에게 강제적으로 느껴지며, 반감과 불신으로 이어져요.

### 3.4 예상하지 못한 순간에 광고가 노출되는 경우

사용 흐름 중 갑작스럽게 등장하는 **전면 광고**.

> 사용 흐름 중 갑작스러운 광고는 몰입을 방해하고, 불쾌한 인상을 남겨요.

### 3.5 CTA 버튼만 보고 다음 행동을 예상할 수 없는 경우

CTA에 화면에서 설명한 가치를 그대로 반복해서 **어떤 행동으로 이어지는지 알 수 없는 경우**.

> CTA는 사용자가 다음에 무엇을 하게 되는지 명확하게 알려주는 장치예요.
> 모호한 버튼 라벨은 전환율 저하로 이어져요.

## 4. 미니앱 브랜딩 가이드

앱인토스 서비스가 토스와 명확히 구분되는 브랜드 경험을 전달할 수 있도록 하는 기준이에요.

### 4.1 브랜드 로고

- 크기: **600×600px 각진 정사각형** (모서리 둥근 형태 불가)
- 반드시 **배경이 있어야** 하며, 라이트/다크 모드 모두에서 잘 보이는 배경 색상 사용
- 로고 자체에 배경이 포함된 경우 600×600px 영역에 꽉 차게 배치
- 노출 위치: 전체탭, 혜택탭, 푸시, 알림, 내비게이션, 브릿지

### 4.2 브랜드 이름

- 특별한 이유가 없다면 **한글**로 작성
  - O: `토스` / X: `Toss`
- 노출 위치: 전체 탭, 혜택 탭, 푸시, 알림, 내비게이션, 브릿지

### 4.3 브랜드 컬러

- 이미 브랜드 컬러가 있다면 그대로 사용
- 없으면 로고에서 가장 많이 사용된 색상을 선택
- 색 대비 기준을 충족하지 못하면 기존 색상을 유지하면서 자동 보정
- 6자리 헥스 코드 사용 (예: `#3182F6`)

### 4.4 내비게이션 바

화면 상단에 고정되는 영역으로, 앱인토스 전용 컴포넌트를 제공.

- 우측 상단 닫기 버튼이 정상적으로 노출·동작해야 함
- Safe Area 영역을 침범하지 않아야 함 (iOS Dynamic Island 포함)

### 4.5 탭바

- 필수 컴포넌트는 아님
- 사용 시 반드시 **토스에서 제공하는 플로팅 형태의 탭바**를 사용해서 직접 구현
- 토스 메인 화면의 기본 하단 탭과 형태가 겹치면 안 됨
- 탭 개수: **최소 2개, 최대 5개**

## 5. 기타 참고사항

### TDS (Toss Design System)

- 모든 컴포넌트를 반드시 TDS로 디자인해야 하는 것은 아님
- 일관된 사용자 경험을 위해 가능한 한 TDS 사용을 권장
- TDS로 해결이 어려운 영역은 자체 개발
- 커스텀 UI보다 **3~5배 빠르게** 개발 완료 가능

### 다크 모드

- 현재 앱인토스 미니앱에서는 **다크 모드를 지원하지 않음**
- 라이트 모드 기준으로 개발/디자인 및 출시

### 폰트

- Toss Products Sans 폰트는 별도로 제공하지 않음
- 토스 프로덕트 산스체를 적용하려면 TDS 컴포넌트를 이용
- Figma에서는 SF Pro를 사용 (앱에서는 자동으로 Toss Product Sans 적용)

### 검수 절차

앱인토스는 4단계 검수를 거침 (평균 2~3일):
1. **운영 검수**: 앱 정보, 노출 정보, 서류 확인
2. **기능 검수**: 오류 없이 정상 작동하는지 확인
3. **디자인 검수**: 토스 UI 가이드 준수 여부 검토
4. **보안 검수**: 개인정보·보안 관련 문제 확인
